# PRD: v0.11.0 — Scenario-driven improvements

**Date**: 2026-05-28
**Status**: draft
**Target release**: v0.11.0 (minor bump; new modes and a new entity)

---

## Executive Summary

A focused round of improvements surfaced by walking the system through five real user scenarios (bootstrap, capture, elaborate, calendar consult, learnings consult). The walkthrough exposed two functional gaps (batch repurpose, qualitative learnings) and three documentation gaps (voice examples, calendar quick-view, the "draft requires idea" design choice). All five ship together as v0.11.0 — a minor bump because two of them add new behavior (`draft --to` mode, `## Aprendizajes` entity), the rest are docs that travel with them.

---

## Problem Statement

### Current state

Five user scenarios were enumerated and mapped to current coverage:

| # | Scenario | Coverage |
|---|---|---|
| 1 | Bootstrap from zero | ✓ `init` covers it |
| 2 | Capture an idea | ✓ `idea` quick + promote |
| 3 | Elaborate a piece of content | ⚠ gaps in batch repurpose; "draft without idea" undocumented |
| 4 | Consult editorial calendar | ⚠ only via full `weekly` ritual |
| 5 | Consult learnings | ✗ not modeled at all |

### Pain points

- **3c — Fan out after publishing**: a typical workflow is "publish newsletter → adapt to LinkedIn + X + blog". Today this is N invocations of `draft --from`, one per target channel. Tedious and easy to forget channels.
- **5 — Qualitative learnings have no home**: SPEC §2 explicitly excluded analytics, but qualitative notes ("this hook worked", "people responded to the personal angle") aren't analytics. They're the data behind the 4-month strategy review, and today they don't exist anywhere.
- **4 — Calendar consult forces a full ritual**: if you just want to see the calendar without doing triage / hygiene / drafting decisions, `weekly` is overkill.
- **3a — "Draft requires idea" is undocumented intent**: a user new to the system reasonably asks "can I draft without going through an idea file?" The answer ("no, by design — the idea is the anchor for cross-channel repurpose later") is not anywhere in user-facing docs.
- **Voice examples — init Paso 7 leaves the user staring at the cursor**: the SPEC §5 voice.md schema has a worked example; init's voice sub-flow doesn't show it during the interview. Same friction we already fixed for strategy's ICP/Goals in v0.10.2 — this is the equivalent fix for voice.

## Goals & Non-Goals

### Goals

1. Make "publish → fan out to N channels" a single command (batch repurpose).
2. Give qualitative learnings a defined home in the data model so the 4-month strategy review has data to draw on.
3. Close the voice-examples gap in init Paso 7 with canonical examples lifted from SPEC §5.
4. Document the "calendar quick-view via conversation" and "draft requires idea" patterns so users aren't surprised.

### Non-Goals

- **Quantitative analytics**. No engagement metrics, no subscriber counts, no open rates. SPEC §2 was right to exclude this; v0.11.0 stays consistent.
- **A `voice` skill**. Voice is still rarely-changing configuration. The fix is examples in the existing init sub-flow, not a new skill. Earmark for dogfooding.
- **A `channels` skill**. Same reasoning — earmark, don't build.
- **A `learnings` skill**. The mechanism is a section in publication files plus a `weekly` hygiene check. Building a dedicated skill for "the user types bullets" is the YAGNI pattern that the post-review caught us doing in v0.10.0.
- **A `--calendar-only` flag for weekly**. Calendar consult via natural-language to Claude Code is the right path; adding a flag is deuda. README documents the pattern.
- **Bypass mode for "draft without idea"**. The two-step (`idea --promote` → `draft`) is intentional. The fix is documenting *why*, not removing the step.

## User Stories

### US-1: Batch repurpose after publishing

**As an** indie hacker who just published a newsletter,
**I want to** invoke a single command to draft adaptations for LinkedIn, X, and the blog,
**So that** I don't have to remember three separate `draft --from` invocations and risk skipping a channel.

Acceptance Criteria:
- [ ] `draft <idea> --from <publication> --to ch1,ch2,ch3` accepts a comma-separated list of target channels.
- [ ] Pre-flight: all target channels exist in the project; abort with a clear list if any missing.
- [ ] For each target channel, the existing single-channel `draft --from` flow runs: read voice + channel + idea + source publication, generate, show, confirm, write, run review checklist.
- [ ] The user confirms each generated draft individually (no "blanket yes"). Confirmation rejection on one draft does not block the others.
- [ ] Final summary lists which target channels resulted in `draft`, which in `ready`, which the user rejected.

### US-2: Capture qualitative learnings on a publication

**As an** indie hacker reviewing what worked last quarter,
**I want to** have a place to write "the hook landed", "subject line was too long", "people replied to the personal angle" on each publication after publishing,
**So that** the 4-month strategy review has real data to contrast against pillars and ICP.

Acceptance Criteria:
- [ ] Publication files gain an optional `## Aprendizajes` H2 section at the end of the body. Free-form bullets; no schema.
- [ ] `weekly` hygiene 3b, when transitioning `ready → published`, prompts: "¿algo que aprendiste de esta? (puedes saltar)". On affirmative, append the bullets the user dictates to the publication file under `## Aprendizajes`. On skip, no section is created.
- [ ] `weekly --cleanup` adds a new hygiene step "Publicaciones sin aprendizajes" that lists `status: published` items older than `learnings_prompt_threshold_days` (default 60) that lack a `## Aprendizajes` section, and offers the user the chance to add them retroactively.
- [ ] `audience-ops-strategy` reads the `## Aprendizajes` sections across publications of each pillar during modo update of bloque 3.3 (Pilares). Surfaces 3-5 recent bullets per pillar AND offers an active prompt: "¿refinar la descripción de este pilar basándote en estos aprendizajes? (refinar / dejar / ver más bullets)". Iteración del pilar es entonces opt-in del usuario, no automática.

### US-3: Voice examples during init

**As a** first-time user defining my voice during `init`,
**I want to** see canonical examples for each section (Atributos, Sí, No, Referencias),
**So that** I have something concrete to react to instead of inventing from nothing.

Acceptance Criteria:
- [ ] `init` Paso 7 (Sub-flujo · Voz y tono) shows a brief example per section, lifted from SPEC §5 voice.md, before asking the user for theirs.
- [ ] Examples are presented as "ejemplo, no obligatorio" — the user is encouraged to deviate.
- [ ] The user can answer "como el ejemplo" and the skill writes the canonical example as-is (for fast bootstrap; refinable later by editing voice.md).

### US-4: Calendar quick-view via conversation

**As a** user who just wants to see the editorial calendar without running a full weekly ritual,
**I want to** know I can ask Claude Code directly,
**So that** I don't invoke `weekly` for a read-only need.

Acceptance Criteria:
- [ ] README "How to use it" gains a subsection (placed near Weekly ritual) explaining: "If you only want to see the calendar without running the full ritual, ask Claude Code: *'show me my editorial calendar from `projects/*/publications/`'*. The state lives in frontmatter; no skill needed for a read-only consult."
- [ ] AGENTS.md adds an equivalent note for non-Claude-Code agents.

### US-5: "Draft requires idea" as documented design decision

**As a** new user asking "can I draft without going through an idea file first?",
**I want to** find the answer (and the reason) in the README,
**So that** I don't waste time looking for a hidden flag that doesn't exist.

Acceptance Criteria:
- [ ] README "Drafting · Idea + channel" section gains a sentence: "The idea file is mandatory by design — it's the anchor that makes cross-channel repurpose coherent later. If you want to draft something timely without first formalizing an idea, run `/audience-ops-idea --promote "<text>"` first; it's a 30-second interview."
- [ ] SPEC §6 strategy table row for `draft` gains a parenthetical: "(idea-slug obligatorio; anchor para repurpose)".

## Functional Requirements

### F1: `draft --to ch1,ch2,...` batch repurpose

**Where**: `.claude/skills/audience-ops-draft/SKILL.md`

**Behavior**:
1. Parse `--to` argument: comma-separated channel ids.
2. Pre-flight: for each target channel id, verify `projects/<slug>/channels/<channel>.md` exists. If any missing, abort listing all missing channels.
3. Run the existing single-channel `draft --from` flow N times sequentially:
   - Read voice + target channel + idea + source publication (read once, reuse where possible).
   - Generate, show, confirm, write, review.
4. Each target gets its own publication file: `projects/<slug>/publications/<idea-slug>-<target-channel>.md`.
5. If a target file already exists, follow the existing "iterate / overwrite / skip" prompt for that one.
6. At the end, render a summary table: target channel | resulting status | path.

**Edge cases**:
- User passes only one channel in `--to`: behaves identically to single-channel `--from` mode.
- User passes target channel that equals source publication's channel: warn ("you're regenerating the source channel"), allow with confirmation.
- A review checklist fails on draft 2 of 3: drafts 1 and 3 still complete normally; draft 2 stays as `draft` with notes.
- User aborts mid-batch (Ctrl-C): drafts already written stay written. Summary lists what was completed.

### F2: `## Aprendizajes` section + weekly integration

**Where**:
- `SPEC.md` §5 (publication schema): document the optional H2 section.
- `.claude/skills/audience-ops-weekly/SKILL.md` (hygiene 3b + cleanup new step).
- `.claude/skills/audience-ops-strategy/SKILL.md` (read for context during staleness warning).
- `config.yaml`: new `learnings_prompt_threshold_days` (default 60).

**Behavior**:
- The section is **optional** structurally and free-form in content. A publication without it is valid; one with empty bullets is also valid (rare).
- Weekly hygiene 3b (ready → published transition) prompts after the status change is confirmed, **only if `prompt_learnings_on_publish: true` in config.yaml (default true)**: "¿algo que aprendiste de esta? (puedes saltar)". On affirmative, capture bullets, append `## Aprendizajes` section to the file. On skip, no section is created. If the flag is `false`, the prompt is suppressed and the user adds aprendizajes later (manually or via cleanup retroactive sweep).
- Weekly cleanup gains a new step (Cleanup 6: "Publicaciones sin aprendizajes") listing published items older than `learnings_prompt_threshold_days` without an `## Aprendizajes` section. Same prompt as 3b, retroactive.
- Strategy skill, in modo update bloque 3.3 (Pilares), surfaces 3-5 recent bullets per pillar from `## Aprendizajes` sections AND offers active iteration: "¿refinar la descripción de este pilar basándote en estos aprendizajes?". Three answers: refinar (entra en sub-interview de ese pilar con los aprendizajes como contexto) / dejar / ver más bullets. Does not block the interview if no learnings exist — sigue mostrando "(no recent learnings recorded for this pillar)" y continúa.

**Edge cases**:
- Publication file has multiple `## Aprendizajes` H2s (user edited badly): treat as one section, concat bullets.
- User dictates aprendizajes that contain a markdown structure (code block, image): preserve as-is (append literal).
- A pillar has zero publications with aprendizajes: strategy skill says "(no recent learnings recorded for this pillar)" and continues.

### F3: Voice examples in init Paso 7

**Where**: `.claude/skills/audience-ops-init/SKILL.md` Paso 7.

**Behavior**:
- Before asking the user for each of the 4 sub-sections (Atributos, Sí, No, Referencias), show the canonical example from SPEC §5 verbatim, labeled "Ejemplo (no obligatorio)".
- After showing, the prompt is the existing one ("Dame 3-5 atributos…") plus a one-liner: "(o di 'como el ejemplo' si te encaja)".
- If the user says "como el ejemplo", the skill writes the canonical example as-is.

**Edge cases**:
- User says "como el ejemplo" for all 4 sections: end result is a voice.md identical to SPEC §5 example. This is intentional — a usable baseline, refinable later by editing voice.md.

### F4: Calendar consult docs

**Where**: `README.md` (How to use it, near Weekly ritual) and `AGENTS.md` (Otros agentes section).

**Behavior**: pure docs. No code changes. The README addition documents two natural agrupaciones to surface the regenerable-views design explicitly:

- **By week** (calendar view): "Muéstrame las publicaciones programadas en `projects/*/publications/`, agrupadas por semana."
- **By status** (kanban view): "Agrupa todas las publicaciones por status (draft / ready / published / overdue) en lugar de por fecha."

Both queries use the same primary state (frontmatter of publication files). The doc explicitly clarifies there is no persisted `calendar.md` or `kanban.md` file — views are computed on demand per SPEC §3 principios 3 + 4.

### F5: "Draft requires idea" docs

**Where**: `README.md` (How to use it, Drafting subsection) and `SPEC.md` §6 (skills table).

**Behavior**: pure docs. No code changes.

## Technical Considerations

### Data model changes

- `SPEC.md` §5 publication schema gains `## Aprendizajes` as an optional H2 section. Frontmatter is unchanged.
- `config.yaml` gains two new keys: `learnings_prompt_threshold_days: 60` (cleanup hygiene threshold) and `prompt_learnings_on_publish: true` (gate for the inline 3b prompt during ready→published transitions).
- Two new entries in `weekly`'s reads-and-writes table.

### Skill changes

| Skill | Change |
|---|---|
| `audience-ops-init` | Paso 7 shows voice examples |
| `audience-ops-draft` | New `--to` mode (multi-channel from a source) |
| `audience-ops-weekly` | Hygiene 3b adds aprendizajes prompt; Cleanup gains step 6 |
| `audience-ops-strategy` | Reads `## Aprendizajes` for staleness-warning context |
| `audience-ops-idea` | No change |

### Versioning

Single bundle as v0.11.0. Justification: F1 (new draft mode) is a minor bump on its own. F2 (new entity) is a minor bump on its own. Bundling them keeps install/update simple (one `npx skills update` brings everything). All 5 SKILL.md `metadata.version` bumped to 0.11.0.

### YAML guard

CLAUDE.md's pre-tag YAML validation step (added in v0.10.1) runs as usual. No new schemas, no new failure modes.

## Success Metrics

This is a personal-MVP repo; success metrics are qualitative and gated on dogfooding.

| Metric | Target | How to measure |
|---|---|---|
| Batch repurpose used in real workflow | At least one publication fanned out to ≥2 channels in a single `draft --to` invocation | Manual check after 4 weeks of use |
| Aprendizajes captured | ≥50% of `published` items have an `## Aprendizajes` section after 8 weeks of use | grep + count |
| Voice friction reduced | First-time user (not Roberto) running `init` produces a non-`_pendiente_` voice.md without asking for help | Observational |
| Calendar consult discoverability | Roberto consults the calendar via conversation instead of `weekly` at least once | Self-report |
| v0.11.0 ships without packaging bugs | YAML validation step blocks any broken frontmatter pre-tag | CLAUDE.md release flow |

These metrics inform whether v1.0.0 ("MVP validated") is reachable.

## Open Questions

All open questions from the initial draft have been resolved during the spec interview:

- ✅ **Strategy iteration of pillars via learnings**: confirmed — F2 now includes an active "¿refinar este pilar basándote en estos aprendizajes?" prompt in modo update, opt-in by the user. Not just read-only context.
- ✅ **Global flag for the on-publish prompt**: confirmed — `config.yaml` gets `prompt_learnings_on_publish: true` (default), so users who find the prompt intrusive can suppress it.
- ✅ **Where to document non-goal "skill candidates" (`voice`, `channels`, `init --add-channel`)**: confirmed — nowhere new. The PRD's Non-Goals + Out of scope sections capture the rationale and the condition for revisiting. A separate `thoughts/backlog.md` would have been overhead without value at this scale; the absence of such a file is itself a design choice.

## Risks

| Risk | Impact | Mitigation |
|---|---|---|
| Batch repurpose feels like "fire and forget" — user clicks through reviews without engaging | Medium | Each target draft requires individual confirmation; the review checklist for each target runs as today. The skill does not have a "skip review for all" shortcut. |
| Aprendizajes section grows unbounded and becomes noise in the publication file | Low | Free-form bullets, no schema; user controls length. weekly cleanup can later add a "extract aprendizajes to a learnings/ archive" mode if real friction surfaces. |
| Voice examples bias users to write voice.md exactly like the example, losing distinctness | Medium | Examples labeled "no obligatorio". The "como el ejemplo" shortcut is explicit (user opts in). README and AGENTS already say voice is yours. |
| "Draft requires idea" doc lands too softly and users still get confused | Low | Two surfaces (README + SPEC §6 parenthetical) covers both the user landing on the README and the agent reading the SPEC. |
| v0.11.0 ships with another packaging bug like v0.10.0 | Low | CLAUDE.md release flow (post-v0.10.1) includes pre-tag YAML validation that aborts on broken frontmatter. The class of bug is mechanically prevented. |

## Out of scope (call-outs)

These were considered and explicitly rejected for v0.11.0:

- **A dedicated `audience-ops-voice` skill.** Voice refinement remains manual editing. Earmarked for dogfooding to validate.
- **A dedicated `audience-ops-channels` skill.** Adding/pausing channels remains manual. Earmarked for dogfooding to validate.
- **A `weekly --calendar` flag.** Conversation with Claude Code suffices for read-only calendar consults. Adding a flag is debt.
- **Engagement / open-rate / analytics integration.** SPEC §2 explicit non-goal; v0.11.0 respects it.
- **A `learnings` skill.** Aprendizajes are bullets the user dictates during a weekly transition. A skill to read 5 lines and write them back is the YAGNI pattern from v0.10.0.

These rejections should be revisited if dogfooding surfaces real friction. Not before.
