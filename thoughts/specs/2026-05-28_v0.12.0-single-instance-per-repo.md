# PRD: v0.12.0 — single-instance-per-repo refactor

**Date**: 2026-05-28
**Status**: draft
**Target release**: v0.12.0 (minor bump with breaking changes — major architectural shift)

---

## Executive Summary

Refactor Audience Ops from a **multi-project monorepo model** (one content repo lists N projects under `projects/<slug>/`) to a **single-instance-per-repo model** (each product code repo gets its own audience-ops instance under `<repo>/audience-ops/`). Removes the `portfolio.yaml` abstraction, the `projects/<slug>/` nesting, and `defaults.project`. The "project" becomes implicitly the host repo; multi-project use = installing the tool in multiple repos. This aligns the system with how indie hackers actually organize their work (per-product code repos) and how `npx skills add` makes the tool naturally land in those repos.

---

## Problem Statement

### Current State (v0.11.x)

The system assumed a **dedicated content repo** (`rbart-ventures-content` is the canonical example in `.gitignore`) that contains:

```
<content-repo>/
├── portfolio.yaml                        ← lists N projects
├── config.yaml
└── projects/
    └── <slug>/                           ← per-project subfolder
        ├── strategy.md
        ├── voice.md
        ├── channels/
        ├── ideas/
        └── publications/
```

`portfolio.yaml` was the index. `projects/<slug>/` was the per-project namespace. `defaults.project` in `config.yaml` selected the active project. Multi-project features (global calendar in weekly, cross-project repurpose in draft, aprendizajes across pillars from multiple projects) lived inside the system.

### Pain Points

1. **Mental model mismatch with how skills.sh distributes**. `npx skills add r-bart/audience-ops` naturally lands in **existing code repos** (e.g., `verxion-platform/`). When the user runs `/audience-ops-init` there, the current system pollutes the repo root with `portfolio.yaml`, `config.yaml`, and `projects/`. This collides visually with product code, can name-collide with existing `projects/` directories, and feels foreign.

2. **The dedicated-content-repo pattern is rare in practice**. Most indie hackers organize per-product: one code repo per product, one website per product, one brand per product. The "one content monorepo for all projects" pattern was an assumption that doesn't match observed behavior.

3. **`portfolio.yaml` is overhead**. In practice it'd hold a single entry for almost everyone. `defaults.project` becomes vestigial when there's only one project to default to.

4. **Multi-project cross-features are speculative**. The system technically supports global calendar across N projects and cross-project repurpose. In practice, neither has been exercised in dogfooding (which hasn't started). They're features built on theory.

5. **`projects/<slug>/` adds a level of nesting**. Every reference in every skill carries the `projects/<slug>/` prefix. Without it, paths shorten and the system becomes more discoverable.

## Goals & Non-Goals

### Goals

1. **Per-project namespacing**: each product code repo gets its own `audience-ops/` folder, fully self-contained.
2. **Eliminate vestigial abstractions**: remove `portfolio.yaml`, `projects/<slug>/` nesting, `defaults.project` config key, and `slug:` frontmatter where unused.
3. **Project identity = host repo directory name**. Zero configuration. No `name:` field, no slug, no metadata duplication.
4. **Cleaner SPEC**: §3 principle 7 ("Multi-proyecto nativo") rewritten as "single-instance, multi-project via multiple repos". §4 structure simplified. §5 schemas updated.
5. **All 5 skills updated** to operate on the new paths and lose the slug-routing logic.
6. **Honest about the trade**: cross-project features (global calendar, cross-project repurpose, cross-pillar aprendizajes across projects) are explicitly removed from this release with reasoning documented.

### Non-Goals

- **Automatic migration of v0.11.x instances**. Anyone with existing content under `projects/<slug>/` does a manual move per release notes. The tool doesn't ship migration logic.
- **Multi-instance aggregation tooling** (a hypothetical `audience-ops-cross --calendar` that walks multiple repos). Earmarked for if dogfooding shows real need. Not in this release.
- **Renaming the `audience-ops/` namespace folder**. Decision taken: `audience-ops/` is the canonical name. Not configurable.
- **Backwards compatibility with v0.11.x file structure**. This is a breaking change, owned explicitly.
- **Display name customization**. Decision taken: project name = repo directory name (e.g., `verxion-platform`). Zero override mechanism.

## User Stories

### US-1: Indie hacker installs audience-ops on an existing product repo

**As a** maker who already has a code repo for my product (`verxion-platform`),
**I want to** run `/audience-ops-init` and have it create a self-contained `audience-ops/` folder inside my product repo,
**So that** my marketing content lives alongside my product code, namespaced cleanly, and persists to GitHub naturally with the rest of the project.

Acceptance Criteria:
- [ ] `/audience-ops-init` invoked from `verxion-platform/` (or any code repo) creates `verxion-platform/audience-ops/` with the full structure (config.yaml, strategy.md, voice.md, channels/, ideas/, publications/).
- [ ] **Does not** create `portfolio.yaml` at repo root.
- [ ] **Does not** create `projects/<slug>/` subdir.
- [ ] If `verxion-platform/audience-ops/` already exists with content, init detects and aborts with a clear message ("audience-ops already initialized here; edit files directly or re-run with `--force`").
- [ ] If the host repo has other code (`package.json`, `src/`, etc.), init does not touch it.

### US-2: Indie hacker runs audience-ops in a dedicated content directory

**As a** maker who prefers a dedicated content directory (no product code mixed in),
**I want to** create a fresh empty directory (e.g., `verxion-content/`), run `/audience-ops-init` there, and have the same `audience-ops/` namespace created inside it,
**So that** the structure is consistent whether I host inside a code repo or in a dedicated directory.

Acceptance Criteria:
- [ ] Behavior is identical to US-1: init creates `<cwd>/audience-ops/` regardless of whether the CWD is empty or has code.
- [ ] No special "dedicated content repo" mode. One model, one path.

### US-3: Existing v0.11.x user updates to v0.12.0

**As a** user who installed Audience Ops at v0.11.x and was about to start dogfooding (i.e., no real content yet),
**I want to** install v0.12.0 and start fresh without confusion about the path change,
**So that** I get the simplified model for my dogfooding.

Acceptance Criteria:
- [ ] v0.12.0 release notes explicitly call out the breaking change.
- [ ] CLAUDE.md release flow documents the v0.11→v0.12 migration (manual file move).
- [ ] If a user has stale `portfolio.yaml` or `projects/<slug>/` from v0.11.x, the v0.12.0 skills do not silently consume them — they look for `audience-ops/` and report "not initialized" if the new structure is absent.

### US-4: Skill operates on the simplified data model

**As an** invoked skill (`weekly`, `strategy`, `draft`, `idea`),
**I want to** read and write files at simplified paths (no `projects/<slug>/` prefix),
**So that** my playbook is shorter, my path references are less error-prone, and I don't have to determine an "active project" before doing anything.

Acceptance Criteria per skill:
- [ ] `init` creates `audience-ops/` structure (see US-1).
- [ ] `idea` operates on `audience-ops/ideas/_inbox.md` and `audience-ops/ideas/<slug>.md`. Removes all `defaults.project` and slug-determining logic.
- [ ] `draft` reads `audience-ops/voice.md`, `audience-ops/channels/<channel>.md`, `audience-ops/ideas/<idea-slug>.md`. Writes `audience-ops/publications/<idea-slug>-<channel>.md`. `--from` and `--to` operate on these paths.
- [ ] `strategy` operates on `audience-ops/strategy.md`. Its lectura previa walks `audience-ops/publications/*.md` (no `projects/*/publications/` glob).
- [ ] `weekly` walks `audience-ops/` only. Calendar view is "this repo's calendar". Hygiene is "this repo's drafts/inbox". Cleanup operates on `audience-ops/publications/archive/<year>/` etc.

## Functional Requirements

### F1: New folder structure (SPEC §4)

Replace SPEC §4 entirely with the new tree:

```
<host-repo>/                              ← any directory (code repo or dedicated)
└── audience-ops/                         ← namespace folder, fixed name
    ├── config.yaml                       ← behaviour thresholds only; NO defaults.project
    ├── strategy.md                       ← positioning, ICP, pillars, goals, anti-topics
    ├── voice.md                          ← voice & tone
    ├── channels/
    │   ├── <id>.md                       ← one file per active channel
    │   └── archive/                      ← paused/retired channels
    ├── ideas/
    │   ├── _inbox.md                     ← quick capture, append-only, dated lines
    │   ├── <slug>.md                     ← structured ideas
    │   └── archive/                      ← killed ideas
    └── publications/
        ├── <idea-slug>-<channel>.md
        └── archive/
            ├── <year>/                   ← published > cleanup_threshold_days
            └── abandoned/                ← abandoned drafts
```

`portfolio.yaml` is **removed**. `projects/<slug>/` is **removed**. The host repo can contain anything alongside `audience-ops/` (code, docs, other tooling) — the namespace is the only contract.

### F2: SPEC §3 principle 7 rewritten

**Before** (v0.11.x):
> "Multi-proyecto nativo. Aun con un solo proyecto, vive bajo `projects/<slug>/`."

**After** (v0.12.0):
> "Single-instance por repo. Cada repo de producto (o directorio dedicado) tiene su propia instancia bajo `<repo>/audience-ops/`. Multi-proyecto se logra **teniendo el tool en varios repos**, no agregando proyectos dentro de un mismo `audience-ops/`. Si necesitas vistas cruzadas entre instancias, se hace fuera de la skill (script externo, agregador manual). Decisión consciente: simplificar el modelo común a costa de las superpowers cross-instance."

### F3: `config.yaml` simplified

```yaml
# config.yaml — cómo se comporta esta instancia de Audience Ops
# Lo lee cualquier skill. Edita a mano si cambias preferencias.

owner:
  name:
  email:

behavior:
  inbox_auto_promote: false
  cleanup_threshold_days: 90
  stale_draft_days: 30
  stale_inbox_days: 30
  strategy_review_months: 4
  paused_archive_threshold_months: 6
  learnings_prompt_threshold_days: 60
  prompt_learnings_on_publish: true
  scheduler: typefully                    # hint informativo para el handoff ready→published
```

Removed:
- `defaults.project`
- `defaults:` block entirely (only `scheduler` survived; moved into `behavior` as the hint it always was)

### F4: `strategy.md` frontmatter simplified

**Before**:
```yaml
---
slug: verxion
last_reviewed: 2026-04-12
pillars: [...]
---
```

**After**:
```yaml
---
last_reviewed: 2026-04-12
pillars: [...]
---
```

`slug:` is **removed**. The project identifier — when needed in user-facing output (calendar render, confirmation messages, prompts) — is derived from `basename(<repo-root>)`. No name configuration, no override.

### F5: All 5 SKILL.md path references updated

| Skill | Path changes |
|---|---|
| `init` | Drastic rewrite. No portfolio bootstrap. No project metadata interview (no slug/name/one_liner/status/url/started). Creates `audience-ops/` structure directly. Owner sub-flow optional (only if config.yaml `owner` is empty). Strategy + voice + channels sub-flows preserved but write to `audience-ops/` not `audience-ops/projects/<slug>/`. |
| `idea` | `projects/<slug>/ideas/_inbox.md` → `audience-ops/ideas/_inbox.md`. Same for structured idea files. Removes "Determinar proyecto" step entirely (no slug to determine). |
| `draft` | `projects/<slug>/voice.md` → `audience-ops/voice.md`. Same for channels, ideas, publications. `--from` and `--to` continue to work but operate within `audience-ops/`. |
| `strategy` | `projects/<slug>/strategy.md` → `audience-ops/strategy.md`. `projects/<slug>/publications/*.md` → `audience-ops/publications/*.md`. Same for idea-to-pillar mapping logic. Frontmatter loses `slug:`. |
| `weekly` | `projects/<slug>/ideas/_inbox.md` → `audience-ops/ideas/_inbox.md`. Same for all paths. Calendar render is "this repo's calendar" (no cross-project aggregation). Removes "filtro `weekly <proyecto>`" feature entirely — there's no other project to filter to. Cleanup 5 (paused projects) is **removed** — no portfolio.yaml means no project status to flip. |

### F6: Cross-instance features explicitly removed

- **weekly global calendar across projects**: gone. Each instance shows its own calendar.
- **draft cross-project repurpose**: removed. `--from` and `--to` operate within the same instance.
- **strategy aprendizajes across projects**: removed. Each strategy reads only its own publications.
- **portfolio-level project lifecycle** (`status: paused → archived`): removed. The instance is alive or not; the host repo lifecycle handles this implicitly.

These removals are **honest about the tradeoff**: cleaner model, fewer abstractions, lose some imagined-but-untested cross-project superpowers.

### F7: Documentation updated

- **SPEC.md**: §3 principle 7 rewritten (F2). §4 structure replaced (F1). §5 schemas updated (F3, F4). §6 skills table paths updated. §10 "Próximos pasos" no longer references multi-project workflows.
- **README.md**:
  - "Repo structure" / "Layout" section: shows the new simplified tree, including the host-repo + `audience-ops/` namespace explicitly.
  - "Install" section: confirms project-scope install (with optional `-g`); explicitly states `audience-ops/` will be created at CWD.
  - "How to use it" / Day 1: notes that the structure goes into `<your-repo>/audience-ops/`.
  - Removes any mention of "multi-project" as a system feature; reframes as "per-repo".
- **AGENTS.md**: same reframe for non-CC agents.
- **CLAUDE.md**: release flow unchanged; add a note about the v0.11→v0.12 model shift for posterity.

## Technical Considerations

### Data model changes (summary)

| Item | v0.11.x | v0.12.0 |
|---|---|---|
| `portfolio.yaml` | exists, lists projects | **deleted** |
| `<repo>/projects/<slug>/...` | per-project namespace | **removed** |
| `<repo>/audience-ops/` | does not exist | **new namespace folder** |
| `config.yaml` `defaults.project` | exists | **removed** |
| `strategy.md` frontmatter `slug:` | required | **removed** |
| Cross-project features | exist (untested) | **removed** |

### Migration (manual)

The breaking change is owned explicitly. Users with v0.11.x content do:

```bash
cd <your-content-repo>
mkdir audience-ops
git mv projects/<slug>/strategy.md audience-ops/strategy.md
git mv projects/<slug>/voice.md audience-ops/voice.md
git mv projects/<slug>/channels audience-ops/channels
git mv projects/<slug>/ideas audience-ops/ideas
git mv projects/<slug>/publications audience-ops/publications
git mv config.yaml audience-ops/config.yaml
git rm portfolio.yaml
# Edit audience-ops/config.yaml to remove `defaults.project` if present
# Edit audience-ops/strategy.md to remove `slug:` line from frontmatter
git commit -m "chore: migrate to v0.12.0 single-instance layout"
```

Per-repo manual move, ~5 minutes. Documented in v0.12.0 release notes.

If a user has multiple projects in one portfolio (rare per usage assumption), they can split into multiple instances:

```bash
mkdir verxion-content audience-ops-marketing
mv projects/verxion/* verxion-content/audience-ops/
mv projects/audience-ops/* audience-ops-marketing/audience-ops/
# Each gets its own config.yaml derived from the shared one
```

### Skill behavior with missing/wrong structure

- `audience-ops/` doesn't exist + CWD is empty: init bootstraps. (Normal first-run.)
- `audience-ops/` doesn't exist + CWD has product code: init bootstraps inside `audience-ops/`. (Normal first-run on a host repo.)
- `audience-ops/` already exists with structure: skills operate on it. (Normal repeat use.)
- `audience-ops/` exists but missing files (e.g., no `strategy.md`): each skill validates its own dependencies and aborts with a clear message ("audience-ops/strategy.md not found; re-run init or create manually").
- Legacy `portfolio.yaml` or `projects/<slug>/` from v0.11.x present: skills **ignore** them entirely. They operate only on `audience-ops/`. If init is invoked, it warns once ("detected v0.11.x layout; see release notes for migration") but doesn't auto-migrate.

### YAML guard

CLAUDE.md's pre-tag YAML validation step (added in v0.10.1) runs as usual on all 5 SKILL.md. No new schemas, no new failure modes for validation.

### Versioning rationale

This is a **minor bump (0.11.x → 0.12.0) with breaking changes**. Justification per CLAUDE.md SemVer policy: SemVer reserves MAJOR for "incompatible API changes". For a markdown-playbook system pre-1.0, breaking changes during 0.x.x are normal — the SemVer pre-1.0 contract is "expect breaks". v1.0.0 remains gated on dogfooding-validated MVP.

If this had landed post-1.0.0, it would have been a 2.0.0 release.

| Risk | Impact | Mitigation |
|---|---|---|
| Existing v0.11.x users with content lose access if they don't migrate before updating | Medium (low population — basically only Roberto who hasn't started dogfooding) | Explicit release notes; skills warn on legacy structure detection; no auto-migration to keep the migration explicit |

That's literally what dogfooding is for. The whole point of the model shift is that it's easier to revise in response to real use. |

## Out of scope (call-outs)

Considered and explicitly rejected for v0.12.0:

- **Auto-migration tooling** (a `audience-ops-migrate` skill or CLI subcommand). Manual `git mv` is good enough at this scale. If a future v1+ release needs auto-migration for a different change, that's a separate scope.
- **A "portfolio" or "ventures" multi-instance aggregator**. Earmarked for "if dogfooding shows need." Not in v0.12.0.
- **Configurable namespace folder name** (not `audience-ops/` but `content/` or `marketing/` or whatever). Fixed name keeps the contract simple and discoverable. If a user really wants different naming, they can rename and adjust paths in their config.yaml — but the skills won't support it natively.
- **Auto-detection of "code repo vs dedicated content dir"** (different default behavior based on host). One model, one path: always `audience-ops/`. Simpler.
- **Display name customization** for the project identifier (a `name:` field in config.yaml). Repo directory name is enough; the user can name their repo as they please.
- **Backwards-compatible read of v0.11.x structure**. Skills don't read `projects/<slug>/` at all in v0.12.0. The model is clean; legacy is migrated explicitly or ignored.

These rejections are recorded so future-me doesn't re-debate.
