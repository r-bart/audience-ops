# PRD: v0.3.0 — per-project dossiers (capture-first pattern)

**Date**: 2026-05-28
**Status**: draft
**Target release**: v0.3.0 (minor — adds new entity + extends one skill)

---

## Executive Summary

Add `audience-ops/projects/<slug>.md` as a documented entity for **capturing per-product context** that informs content creation: a "qué es / audiencia / story log / ángulos abiertos" dossier. Establish a slug-prefix convention so ideas can be routed to a project (e.g. `bl-cloudflare-migration` → Brakinglab dossier). Extend `audience-ops-draft` to auto-read the dossier when an idea's slug has a known prefix. Capture-first design: the real bottleneck for indie hackers managing N parallel products is generating publishable material, not routing it.

---

## Problem Statement

### Current state (v0.2.0)

The single-instance-per-repo model works for indie hackers building in public from a personal voice. But it has a gap: **when one writer maintains 3-5 projects in a single instance** (Roberto's actual case), the system doesn't have a place for **per-project context**.

Today:
- `strategy.md` is one-instance-wide. Pillars apply across all projects.
- `voice.md` is one-instance-wide. Same voice everywhere.
- `channels/` is one-instance-wide. Channels are shared.
- `ideas/` has slug + pillar + channels in frontmatter — but no link to "which product is this idea about".

Workarounds that emerged during dogfooding setup:
- Hashtags inline (`#brakinglab`, `#verxion`) in idea bodies.
- Manual `audience-ops/projects/<slug>.md` dossiers created ad-hoc by another agent during this session.
- Slug-prefix convention (`bl-`, `vx-`, `aop-`) sketched as visual grouping.

### Pain points

1. **Capture friction is the real bottleneck, not routing**. The hardest question isn't "where does this idea go?" — it's "what do I have to say?". Indie hackers stare at blank pages because nothing reservoired the decisions/metrics/bugs/conversations that would be publishable material if surfaced.

2. **No per-project context for `draft`**. When drafting a piece tagged for Brakinglab, the draft skill doesn't know Brakinglab's stack, audience, recent decisions, or open story arcs. The writer pastes this context manually each invocation.

3. **Stories die in head between decision-moment and write-moment**. A debugging session you'd love to write up disappears 48 hours later because there was no fast capture surface. The inbox captures ideas-already-formed; nothing captures raw events.

4. **"Where am I at" across N projects is opaque**. Without a per-project surface, knowing "what's the current Verxion narrative state? what have I shipped lately?" requires reconstructing from memory or git logs.

## Goals & Non-Goals

### Goals

1. **Document the dossier entity** as a first-class part of the data model — make it discoverable and useful instead of ad-hoc.
2. **Establish the slug-prefix convention** as documented contract, not folklore.
3. **Bootstrap dossiers via `init`**: optional sub-flow during instance creation to declare 1+ projects up front with scaffolded dossiers.
4. **Add-project flow** for existing instances: `init --add-project` runs only the projects sub-flow to append new dossiers without re-initializing.
5. **Wire dossier into `draft`** so the writer doesn't repeat the manual context-paste.
6. **Capture-first surface**: the `## Story log` and `## Ángulos abiertos` sections give writers a place to dump raw material before it becomes ideas.
7. **Preserve frontmatter-mínimo principle**: dossier frontmatter only carries what skills need to parse; the rest is prose.
8. **Backwards-compatible with v0.2.0**: existing instances without dossiers continue to work unchanged.

### Non-Goals

- **A `audience-ops-dossier` dedicated skill** for edit/delete/archive of dossiers. Editing markdown by hand is fine; archiving = `git mv`. Creating a skill for "write text in a file" is the YAGNI pattern this project has caught itself doing repeatedly. (`init --add-project` is in scope because it scaffolds the structure; editing is not.)
- **Extending `weekly` to surface story-log entries**. Earmarked for evidence-based v0.4.x: only if dogfooding shows story logs accumulate faster than they get promoted to ideas.
- **Extending `strategy` to consume dossier data**. Same earmark.
- **Enforcing slug-prefix on ideas**. Optional convention. Ideas without a prefix are valid; the dossier-read is skipped silently.
- **Auto-detection of prefix → dossier from filename**. Explicit `slug_prefix:` field in dossier frontmatter is the only mapping mechanism (robust, explicit, refactorable).
- **A formal "projects" entity in `portfolio.yaml`-style schema**. v0.2.0 killed `portfolio.yaml`; the dossier is just markdown files under `audience-ops/projects/`. No registry.
- **Tags as frontmatter field on ideas**. The dossier link via slug-prefix replaces the need for a `project:` field. Hashtags inline remain valid for cross-cutting attributes (`#evergreen`, `#data-heavy`).
- **Cross-instance features**. v0.2.0 single-instance contract stands.
- **Restoring multi-project portfolio semantics from v0.11.x**. The dossier is per-project context within ONE instance, not a portfolio of N independent instances. `init` asking about projects is **scaffolding for the dossier pattern**, NOT a return to portfolio.yaml. Frontmatter remains single-instance; pillars stay one-list-wide; voice stays one-file-wide.

## User Stories

### US-1: Indie hacker captures a publishable event without breaking flow

**As an** indie hacker who just had a debugging insight while shipping Brakinglab,
**I want to** append a one-liner to `audience-ops/projects/brakinglab.md` `## Story log`,
**So that** when I sit down to write next week, the material is there waiting instead of fading from memory.

Acceptance Criteria:
- [ ] `audience-ops/projects/brakinglab.md` exists as a plain markdown file with `## Story log` section.
- [ ] Adding an entry is a one-line edit (manual or via `> "append to Brakinglab story log: …"` conversation).
- [ ] No skill invocation required for capture — the file is the surface.
- [ ] SPEC documents the `## Story log` and `## Ángulos abiertos` sections as canonical structure.

### US-2: Writer drafts a piece bound to a project, gets context for free

**As a** writer drafting `bl-cloudflare-migration` for newsletter,
**I want to** invoke `/audience-ops-draft bl-cloudflare-migration newsletter` and have the skill auto-read `audience-ops/projects/brakinglab.md`,
**So that** the generated draft references Brakinglab's actual stack, audience, and recent story-log entries without me re-explaining context each time.

Acceptance Criteria:
- [ ] `audience-ops-draft` SKILL.md documents the dossier-read step.
- [ ] When the idea's slug starts with a prefix declared in any `audience-ops/projects/<slug>.md` frontmatter (`slug_prefix:` field), draft reads that dossier and includes its content in the generation context.
- [ ] When no dossier prefix matches, draft proceeds as today (no error, no warning — silent fallback).
- [ ] Multiple dossiers declaring the same prefix → warn the user, use the first match alphabetically.
- [ ] Idea-slug without prefix (e.g. `general-essay-on-x`) → draft proceeds without dossier.

### US-3: Writer formalizes the idea-to-project binding via slug

**As a** writer promoting an inbox entry to a structured idea about Verxion,
**I want to** name the slug with the `vx-` prefix declared in Verxion's dossier,
**So that** anyone (me, future me, an agent) reading `audience-ops/ideas/vx-*.md` can immediately see which project each idea belongs to.

Acceptance Criteria:
- [ ] SPEC documents the slug-prefix convention (alphanumeric prefix + `-` + descriptive slug).
- [ ] The convention is **suggestive, not enforced**. `idea --promote` does NOT validate prefixes against existing dossiers.
- [ ] `audience-ops-idea --promote` SKILL.md gains a sentence: "If the project context is clear, prefixing the slug with the project's `slug_prefix` (e.g. `bl-`, `vx-`) helps later routing. Optional."

### US-4: Writer reads "where am I at" on a project in seconds

**As an** indie hacker context-switching to Verxion after a week on Brakinglab,
**I want to** open `audience-ops/projects/verxion.md` and see the current narrative state: what is Verxion, who it's for, recent events, open story arcs,
**So that** I can resume content work on Verxion without 30 minutes of "what was I going to write about next?".

Acceptance Criteria:
- [ ] The dossier structure (`## Qué es`, `## Audiencia`, `## Story log`, `## Ángulos abiertos`) is documented in SPEC.
- [ ] Writers maintain story log + ángulos abiertos manually (no skill writes to them automatically).
- [ ] Convention to keep story log dated and reverse-chronological (newest at top) is documented.

### US-5: Indie hacker bootstraps the instance with one or several projects from day one

**As a** new user running `/audience-ops-init` who already knows they're going to manage 1 or N projects in this instance,
**I want to** declare those projects during bootstrap and have init create scaffolded dossiers ready to fill,
**So that** I don't start from zero with the dossier pattern and the slug-prefix convention is set up coherently from the beginning.

Acceptance Criteria:
- [ ] `init` Paso 7 (new) asks: "¿quieres declarar proyectos ahora? (1+; opcional — puedes añadirlos después)".
- [ ] If the user lists one or more projects, init asks for: name, `slug_prefix` (suggesting a 2-3 char prefix derived from the name), URL (optional), short description (optional).
- [ ] init writes `audience-ops/projects/<name-kebab-case>.md` per project with the canonical schema (frontmatter + empty `## Qué es`, `## Audiencia`, `## Story log`, `## Ángulos abiertos` sections).
- [ ] If the user declines, no `audience-ops/projects/` directory is created. Backwards compatible with v0.3.0 instances that opt out.
- [ ] Resumen final mentions the dossiers created and the prefix convention.

### US-6: Writer adds a new project to an existing instance

**As an** indie hacker who started with 2 projects and is now shipping a third,
**I want to** add a new project dossier to the existing instance,
**So that** the new project gets the same scaffolded structure as the original ones, without me copying templates by hand.

Acceptance Criteria:
- [ ] Re-invoking `/audience-ops-init` inside an already-initialized instance (where `audience-ops/config.yaml` exists) detects the state and offers a sub-mode "add project". This replaces the current abort-and-exit behavior **only when the user opts in to add-project mode**; otherwise, init still aborts to protect the instance.
- [ ] The add-project sub-mode runs the same project interview as US-5 and writes a new dossier.
- [ ] Editing or archiving existing dossiers is **manual** (open the file or `git mv` to `audience-ops/projects/archive/`). No skill flow for edit/archive in v0.3.0 — earmarked for v0.4.x if dogfooding asks.

## Functional Requirements

### F1: Dossier entity (`audience-ops/projects/<project-slug>.md`)

**Where**: New schema in SPEC §5. New directory in §4 tree.

**Schema** (loose, frontmatter mínimo):

```markdown
---
name: brakinglab                          # human display name
slug_prefix: bl                           # REQUIRED if you want draft to auto-bind ideas
url: https://brakinglab.com               # optional
stack: [astro, cloudflare-workers, d1]    # optional
status: monetized                         # optional; free-form (building/monetized/paused/archived/...)
last_updated: 2026-05-28                  # optional, manually maintained
---

## Qué es

Una frase clara de qué es el proyecto.

Párrafo opcional con más contexto.

## Audiencia

Quién se beneficia. Casos de uso reales.

## Story log

Reverse-chronological. Una línea por evento publicable:

- 2026-05-28 · Decisión: pasé de Vercel a Cloudflare Workers. $X → $Y, latencia Z.
- 2026-05-20 · Métrica: primer cliente de pago. €Y MRR.
- 2026-05-12 · Bug raro: hidratación de Astro rompía en Safari iOS por …

## Ángulos abiertos

Material del story log que podría madurar a idea estructurada:

- [ ] Justificar Cloudflare vs Vercel con números → `behind-the-build`
- [ ] Por qué Astro encaja mejor que Next aquí → `design-engineering`
```

**Rules**:
- Only `slug_prefix:` is read by skills. Everything else is for human consumption.
- The directory `audience-ops/projects/` is created lazily — if it doesn't exist, no dossiers exist, no behavior changes.
- File naming convention: `<project-slug>.md` where `project-slug` matches the project's identity (kebab-case). The filename and `slug_prefix:` field are independent — the filename is for humans, the field is for the skill.

### F2: Slug-prefix convention for ideas

**Where**: SPEC §4 (folder conventions) + §5 (ideas schema).

**Behavior**:
- Convention: `<prefix>-<descriptive-slug>` (e.g. `bl-cloudflare-migration`, `vx-cardio-rmssd`, `aop-portfolio-killed`).
- Convention is **optional and unvalidated**. An idea named `general-essay-on-discipline` is valid; it just won't auto-bind a dossier.
- No frontmatter field added to ideas. The prefix is part of the slug, derivable by string-split on first `-`.
- Multiple ideas can share a prefix (that's the whole point).

### F3: `audience-ops-draft` extension — auto-read matching dossier

**Where**: `.claude/skills/audience-ops-draft/SKILL.md` Lectura previa section + Paso 2 (Componer contexto).

**Behavior**:

In **Lectura previa**, add as step 0 (before reading voice/channel/idea):
- List `audience-ops/projects/*.md` (excluding `archive/`).
- Parse each frontmatter; build a map `slug_prefix: <dossier-path>`.
- If idea-slug split at first `-` yields a prefix that maps to a dossier, mark that dossier for inclusion.

In **Paso 2 · Componer contexto**, add a new bullet at the top:
- **Project dossier** (if matched): full content of `audience-ops/projects/<project-slug>.md`. Use as context for product references (stack, audience, story log) — but the **idea's hook/ángulo and the channel's format still take priority** when they conflict.

**Edge cases**:
- No `audience-ops/projects/` directory → skip step 0 silently. Backwards compatible with v0.2.0 instances.
- Empty `audience-ops/projects/` directory → same.
- Idea-slug with no `-` (e.g. `essay`) → no prefix to extract; skip.
- Prefix maps to nothing → proceed without dossier; do not warn (could be intentional — writer might use prefix-like slugs for other reasons).
- Two dossiers with same `slug_prefix:` → warn ("Found 2 dossiers with prefix `bl-`: ... Using first match alphabetically: ..."), use first.
- Dossier exists but has no `slug_prefix:` field → silently ignored. Won't bind to any idea. (Allows draft-only dossiers without prefix machinery.)

### F4: `audience-ops-init` extension — sub-flow for project dossiers + add-project mode

**Where**: `.claude/skills/audience-ops-init/SKILL.md`.

**Two sub-changes**:

**4a — New Paso 7 · Sub-flujo · Proyectos (opcional)** between current Paso 6 (Canales) and Paso 7 (Resumen final, which becomes Paso 8).

After channels are set up, ask the user:

> "¿quieres declarar proyectos ahora? (1+; opcional — puedes añadirlos después con `/audience-ops-init --add-project`). Útil si vas a usar esta instancia para varios productos (ej. brakinglab, verxion, audience-ops, blog personal)."

If yes, for each project the user names:
- **name**: human display (ej. "Brakinglab").
- **slug_prefix**: 2-3 char prefix; sugerir derivado del nombre (`Brakinglab` → `bl`, `Verxion` → `vx`). Validar único entre los dossiers que se van creando.
- **url** (opcional).
- **descripción corta** (opcional, 1 frase para `## Qué es`).

Crear `audience-ops/projects/<name-kebab-case>.md` con la plantilla:

```markdown
---
name: <name>
slug_prefix: <prefix>
url: <url>                  # omitir si no se dio
status: building            # default; el usuario lo edita después
last_updated: <YYYY-MM-DD de hoy>
---

## Qué es

<descripción si se dio; vacío si no>

## Audiencia

<vacío inicialmente; el usuario lo rellena después>

## Story log

<!-- Reverse-chronological. Una línea por evento publicable. -->

## Ángulos abiertos

<!-- Material del story log que podría madurar a idea estructurada. -->
```

Si el usuario declina, omitir el paso. No crear `audience-ops/projects/`.

**4b — Add-project sub-mode**:

Update **Lectura previa** step 1 of init:
- Currently: "Si `audience-ops/config.yaml` existe, ABORT."
- New: "Si `audience-ops/config.yaml` existe **y la invocación incluye `--add-project`**, entrar en sub-modo add-project (saltar a Paso 7 · Sub-flujo · Proyectos). Si existe y no se pasó la flag, ABORT como antes."

In add-project mode, the only step run is Paso 7 (proyectos). One or more new dossiers get created. Then jump to Paso 8 (Resumen final) which lists only the new dossiers added.

**Edge cases**:
- `--add-project` invoked when `audience-ops/` doesn't exist → error: "no audience-ops instance found; run `/audience-ops-init` first".
- `slug_prefix` collision with existing dossier → re-prompt with suggestion.
- Project name collision with existing dossier filename → re-prompt with disambiguation suggestion (`brakinglab-v2`, etc.).

### F5: `audience-ops-idea --promote` minor doc tweak

**Where**: `.claude/skills/audience-ops-idea/SKILL.md` Paso 2 (Interview de estructura).

**Behavior**: in the **Slug** bullet, append one sentence:
> "Si la idea pertenece claramente a un proyecto con dossier en `audience-ops/projects/`, usa su `slug_prefix` al principio del slug (ej. `bl-...`, `vx-...`). Convención opcional pero recomendada — permite a `draft` cargar el dossier automáticamente."

No interview change. No validation. Pure suggestion.

### F6: SPEC + README + AGENTS updates

- **SPEC §4**: tree gains `projects/` directory inside `audience-ops/` with one-line description.
- **SPEC §5**: new section "`audience-ops/projects/<slug>.md` (opcional, dossier por proyecto)" describing schema + use.
- **SPEC §6 skills table**: `draft` row updated to include "+ dossier de proyecto si existe y prefix coincide" in the Lee column.
- **README "Layout"**: mention `projects/` as optional with one-liner.
- **README "How to use it"**: add a small section "Capture story logs per project" explaining the pattern.
- **AGENTS.md**: brief mention that agents reading the instance should expect optional `projects/<slug>.md` dossiers.

## Technical Considerations

### Data model

| Item | v0.2.0 | v0.3.0 |
|---|---|---|
| `audience-ops/projects/` | Does not exist | Optional directory with `<slug>.md` dossiers |
| Dossier frontmatter | n/a | Only `slug_prefix:` is parsed; rest is for humans |
| Ideas frontmatter | unchanged | unchanged (no new field) |
| Slug-prefix convention | undocumented (folklore) | Documented in SPEC §4 + §5 |
| `init` flow | strategy + voice + channels | + optional projects sub-flow |
| `init --add-project` | n/a | New sub-mode for existing instances |
| `draft` reads | voice, channel, idea, strategy | + optional dossier |

Net: one new entity, fully optional, frontmatter mínimo. Two skills extended (init, draft), one minor doc tweak (idea).

### Backwards compatibility

- Instances without `audience-ops/projects/` continue to work identically.
- Ideas without slug prefix continue to work identically.
- The dossier-read in draft is silent when there's nothing to bind.
- No migration required from v0.2.0 → v0.3.0.

### YAML guard

CLAUDE.md release flow includes pre-tag YAML validation. F1's dossier frontmatter is YAML; if a user writes broken YAML in a dossier, draft fails to parse and **abort the dossier read** (do not block the rest of the draft). Surface the error to the user.

### Discovery cost

`draft` will list `audience-ops/projects/*.md` on every invocation. For N dossiers (expected 3-10 for the ICP), this is cheap. If N > 50, consider caching — but that's premature optimization. Document the soft cap.

## Open Questions

All resolved during writing:

- ✅ **Naming of directory**: `projects/` (user decision — "builders say projects, not dossiers").
- ✅ **Prefix → dossier mapping mechanism**: explicit `slug_prefix:` field in dossier frontmatter. Filename is for humans, not skills.
- ✅ **Dossier schema strictness**: loose. Only `slug_prefix:` is parsed by skills. The rest (name, url, stack, status, last_updated) is convention, not contract.
- ✅ **Idea frontmatter changes**: none. Prefix is part of slug, derivable. Frontmatter mínimo.
- ✅ **Scope**: draft extension only. Idea gets a one-sentence doc nudge. Weekly + strategy are explicitly out for v0.3.0.
- ✅ **Enforcement**: optional, unvalidated. Suggestions in docs; zero hard requirements.

## Out of scope (explicit call-outs, for future-me)

These were considered and explicitly rejected for v0.3.0:

- **`audience-ops-dossier` skill**. Editing markdown by hand is fine. No interview needed.
- **`weekly` reads story logs**. Earmarked for v0.4.x if dogfooding shows story logs accumulate without conversion to ideas.
- **`strategy` reads dossiers**. Same earmark. Strategy should focus on cross-project narrative (pillars, ICP). Dossiers are per-project context.
- **`tags:` field in idea frontmatter for cross-cutting attributes**. Hashtags inline in body remain the supported pattern for `#evergreen`, `#data-heavy`, etc. The dossier solves the project-binding case; tags solve attribute-binding (different problem).
- **Enforced slug-prefix validation in `idea --promote`**. Pure suggestion. Validating prefixes against existing dossiers couples the skill to dossier-state in ways that hurt during early/sparse use.
- **Cross-instance dossiers** (one dossier referenced from multiple instances). Out of v0.2.0 single-instance contract.
- **Dossier-to-channel preferred mapping** (e.g. `verxion.md` declares "newsletter is my primary channel"). Earmarked. Could simplify `idea --promote` channel selection if dogfooding asks for it.

These rejections are documented so the next PRD doesn't redebate them blind.
