## Overview

Implement v0.11.0 — five improvements surfaced by walking real user scenarios. Two functional (batch repurpose, qualitative learnings), three documentation (voice examples in init, calendar consult patterns, "draft requires idea" design decision). All ship together as a single bundled minor release.

Traceable to spec User Stories:

- [ ] US-1: `draft --to ch1,ch2,...` batch repurpose with per-target review and final summary.
- [ ] US-2: `## Aprendizajes` optional H2 in publications; weekly hygiene 3b prompt (gated by config); weekly cleanup retroactive sweep; strategy reads aprendizajes per pillar and offers active iteration in modo update.
- [ ] US-3: `init` Paso 7 shows canonical voice examples per sub-section; "como el ejemplo" shortcut.

- [ ] US-5: README + SPEC §6 explicitly document that draft requires an idea-slug as design choice (anchor for repurpose).

## Approach Analysis

### Option A: Phased by feature, ordered by complexity ascending

**Description**: Each phase completes one feature in full (SPEC + config + skills + docs as needed). Order from simplest (pure docs) to most complex (aprendizajes integration touching SPEC + config + 3 skills).

**Pros**:
- Each phase is independently verifiable.
- If the final phase (F2) hits issues, F1/F3/F4/F5 are already integrated.
- Natural complexity ramp keeps cognitive load manageable.

**Cons**:
- Mixes "schema change" and "behavior change" within the same phase for F1 and F2.

**Complexity**: Medium.

### Option B: Phased by layer (schema → behavior → release)

**Description**: Phase 1 all SPEC + config changes, Phase 2 all docs, Phase 3 all skill behavior, Phase 4 release.

**Pros**:
- Clean architectural separation; each phase has a single concern.
- All "contract" changes (SPEC, config) land before any "behavior" changes — defensive against contract drift mid-implementation.

**Cons**:
- F2's pieces are scattered across 3 phases — harder to reason about as a single feature.
- A bug in F2 surfaced in Phase 3 might require revisiting Phase 1 work.

**Complexity**: Medium.

### Recommendation

**Option A**. Each phase delivers a self-contained, verifiable feature. We can stop mid-implementation if dogfooding insights emerge from earlier phases without leaving half-feature state. The contract-vs-behavior separation in Option B is sound architecturally but adds coordination overhead this MVP doesn't need.

### Phase 3: F1 — `draft --to` batch repurpose

#### Task 3.1: Add `--to` batch mode to draft skill

**File**: `.claude/skills/audience-ops-draft/SKILL.md`

**Three sub-changes** in this file:

**3.1a** — Update the "## Entradas" section: add an `--to` flag description after the existing `--from` description.

**Insertion content**:

```markdown
- Opcional **--to `<ch1,ch2,...>`**: lista comma-separated de canales destino para **batch repurpose**. Cuando se usa con `--from`, transforma la publicación origen a cada uno de los canales destino en una sola invocación, generando un fichero por canal. Sin `--from`, batch desde una idea pelada (mismo behavior, N drafts en una invocación). Pre-flight: si algún canal listado no existe en el proyecto, abortar antes de generar nada.
```

**3.1b** — Add a new top-level section "## Modo batch (`--to`)" between "## Modo repurpose (`--from`)" and "## Escritura · Ficheros creados o modificados".

**Insertion content**:

```markdown
## Modo batch (`--to`)

Cuando se pasa `--to <ch1,ch2,...>` (combinable con `--from`):

### Pre-flight

1. Parsear la lista comma-separated en N canal-ids.
2. Para cada `<channel>`, verificar que existe `projects/<slug>/channels/<channel>.md`. Si **alguno** falta, abortar listando los que faltan. Sin esto se genera basura.
3. Validar que ningún canal destino aparece duplicado (warning, no error).
4. Si `--from` también se pasa: leer la publicación origen una vez. Si no, leer la idea y proceder como modo normal × N.

### Generación secuencial

Para cada canal destino, en orden de aparición:

1. Ejecutar el flujo de generación de modo normal (lectura de voice + canal + idea + opcional origen → componer contexto → generar → mostrar → confirmar → escribir → review).
2. Cada draft escribe `projects/<slug>/publications/<idea-slug>-<channel>.md`.
3. Si el fichero ya existe, aplica el flujo existente "sobreescribir / iterar / abandonar" para ese canal individual. No se aplica blanket-yes para el batch.
4. Si la review checklist falla en un canal individual, ese fichero queda como `draft` con notas; el siguiente canal sigue su curso sin verse afectado.

### Particularidades

- **Rechazo individual del usuario** (decir "no" al confirmar un draft generado): el fichero NO se escribe, el siguiente canal procede.
- **Abort mid-batch** (usuario interrumpe): los drafts ya confirmados/escritos quedan; el resumen final refleja qué se completó.
- **Canal destino == canal del origen** (en modo `--from`): warning ("estás regenerando el canal origen — ¿continúas?"), permitir tras confirmación. Útil cuando se quiere reescribir el origen con un ángulo distinto.

### Resumen final

Tras el batch, renderizar tabla:

| Canal destino | Resultado | Path |
|---|---|---|
| linkedin | ready | `projects/<slug>/publications/<idea>-linkedin.md` |
| x | draft (review failed: subject) | `projects/<slug>/publications/<idea>-x.md` |
| blog | rechazado por el usuario | (no escrito) |

```

**3.1c** — Update "## Errores y casos límite" with `--to`-specific edge cases.

**Insertion content** (append to the existing list):

```markdown
- **Modo `--to` con canal inexistente**: pre-flight aborta listando todos los canales faltantes. No se genera nada.
- **Modo `--to` con un solo canal**: behavior idéntico a modo single-channel; el flag se acepta por simetría.
- **Modo `--to` mezclado con `--from <publication-path>` cuyo canal está en la lista**: warning de regeneración del origen, permitir con confirmación explícita.
- **Modo `--to` con interrupt mid-batch**: los drafts confirmados quedan escritos; el resto se omite; resumen lista lo completado.
```

**File**: `SPEC.md`

**Where**: §5, in the publication schema section (`projects/<slug>/publications/<idea-slug>-<channel>.md`), after the frontmatter example block.

**Insertion content** (append to the existing description of the publication schema):

```markdown
### `## Aprendizajes` (opcional, post-publicación)

Cuando una publicación llega a `status: published`, puede acumular una sección opcional `## Aprendizajes` al final del cuerpo, con bullets libres:

```markdown
[cuerpo de la publicación]

## Aprendizajes

- El hook con dato concreto funcionó mejor que el ángulo personal.
- Subject line tenía 53 chars — algunos clientes lo cortaron en preview.
- Una respuesta nominal: "@user" preguntó si esto aplica a Zone 2.
```

Reglas:
- **Estructura libre**: bullets en plain text, sin schema. Se valoran "qué funcionó / qué no / para la próxima" pero la forma exacta queda al usuario.
- **Opcional**: una publicación published sin `## Aprendizajes` es válida.
- **Append-only en la práctica**: añade bullets a lo largo del tiempo si emergen nuevos aprendizajes (ej. una respuesta tardía).
- **Lo leen**: `weekly` (para hygiene retroactiva en cleanup) y `strategy` (para surface y posible iteración de pilares en modo update).
- **No es analítica**: aprendizajes son qualitativos. Engagement / open-rate / subscribers siguen out-of-scope per §2.
```

## Risk Analysis

### Edge Cases (already covered in skill text but worth restating)

- [ ] `draft --to` with one missing channel: pre-flight aborts before generating.
- [ ] `draft --to` with the same channel as `--from` source: warning + confirm.
- [ ] `draft --to` interrupt mid-batch: confirmed drafts persist, summary lists completed.
- [ ] Aprendizajes prompt when `prompt_learnings_on_publish: false`: silently skipped.
- [ ] Strategy modo update without any aprendizajes: shows "(no hay aprendizajes registrados todavía)" and continues normally.
- [ ] Cleanup 6 with publication that gets aprendizajes mid-loop (concurrent edit): re-read frontmatter is implicit per existing weekly cleanup pattern.
