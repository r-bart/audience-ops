# Implementation Plan: v0.3.0 — per-project dossiers (capture-first)

**Date**: 2026-05-28
**Status**: Draft
**Spec**: `thoughts/specs/2026-05-28_v0.3.0-project-dossiers.md`

---

## Overview

Add `audience-ops/projects/<slug>.md` as documented entity with story-log + ángulos-abiertos sections. Document slug-prefix convention. Extend `init` with a projects sub-flow + `--add-project` mode. Extend `draft` to auto-read matching dossier when an idea-slug has a known prefix. One-sentence tweak to `idea --promote`. Backwards compatible with v0.2.0.

## Requirements

Traceable to PRD User Stories:

- [ ] US-1: Story log + ángulos abiertos sections documented and writable by hand.
- [ ] US-2: `draft` auto-reads dossier when idea-slug prefix matches a `slug_prefix:` field.
- [ ] US-3: Slug-prefix convention documented; `idea --promote` suggests it.
- [ ] US-4: Dossier structure (qué es / audiencia / story log / ángulos abiertos) is canonical and documented.
- [ ] US-5: `init` Paso 7 (nuevo) ofrece declarar 1+ proyectos durante bootstrap.
- [ ] US-6: `init --add-project` añade dossiers a instancias existentes sin re-inicializar.

---

## Approach Analysis

### Option A: Contract-first (SPEC → init → draft → docs → release)

**Description**: SPEC.md adds the dossier entity + slug-prefix convention as documented schema first. Then init implements the bootstrap + add-project flows. Then draft implements the dossier-read. Then docs (README, AGENTS, idea minor tweak). Then release.

**Pros**:
- SPEC defines the data contract once; init and draft both reference the same schema (slug_prefix field, dossier sections).
- Mirrors v0.2.0 plan structure — pattern proven.
- Reduces "is this the right field name?" guesswork during skill edits.

**Cons**:
- SPEC update is one big edit; if a field name is wrong, downstream tasks consume it.

**Complexity**: Low–Medium.

### Option B: Skill-first (define behavior in skills, document in SPEC after)

**Description**: Write init's sub-flow + draft's dossier-read first; let SPEC catch up afterward.

**Pros**: Faster start.

**Cons**: Skills end up referring to schema details ad hoc; SPEC becomes documentation-after-the-fact, which has caused drift in this codebase before.

**Complexity**: Medium.

### Recommendation

**Option A**. Same reasoning as v0.2.0: for markdown playbooks, SPEC is the contract, skills are the implementation. Contract first. Especially important because two skills (init and draft) both reference `slug_prefix:` semantics — they must agree on the field name and behavior, which means a single source of truth defined upstream.

## Audiencia

Quién se beneficia, casos de uso reales.

<!-- Reverse-chronological. Una línea por evento publicable. -->

- **Sólo `slug_prefix:` se parsea** por las skills. El resto del frontmatter (name, url, stack, status, last_updated) es para humanos.

```markdown
### Capture per project · Story log + ángulos abiertos *(v0.3.0+, optional)*

If your instance manages content for multiple products (e.g., one indie hacker, 4 brands), declare each project during `init` (or later with `init --add-project`) and you get scaffolded dossiers in `audience-ops/projects/`.

Each dossier:

- Lives as plain markdown — you edit it by hand.
- Carries a `slug_prefix:` (e.g. `bl`, `vx`, `aop`) — ideas prefixed accordingly bind to it.
- Holds four sections: `## Qué es`, `## Audiencia`, `## Story log`, `## Ángulos abiertos`.

**Story log** is the capture surface for raw events (decisions, metrics, bugs, conversations) that may become publishable later. **Ángulos abiertos** is the "this could be an idea someday" buffer.

When you draft an idea whose slug starts with a known prefix (`/audience-ops-draft bl-cloudflare-migration newsletter`), `draft` reads the matching dossier automatically and includes its content as generation context. No more pasting "remember Brakinglab is on Cloudflare..." into every draft prompt.

**Manually editing**: dossiers are markdown files. Use any editor, or talk to your agent (*"append to brakinglab dossier story log: ..."*). No skill writes to story log or ángulos abiertos automatically — it's your reservoir.

**Optional**: instances without dossiers continue to work identically to v0.2.0. Declare projects only if you want the dossier pattern.
```

## Risk Analysis

### Edge cases (covered in skills text)

- [ ] `--add-project` on non-initialized instance: error (init).
- [ ] Slug-prefix collision in dossiers: warn-and-use-first (draft).
- [ ] Idea-slug without `-`: no prefix to extract; draft proceeds (draft).
- [ ] Dossier with broken YAML: warn + skip dossier; don't block draft (draft).
- [ ] No `audience-ops/projects/` directory: silent skip (draft, idea).
- [ ] Slug-prefix collision during init add: re-prompt with suggestion (init).

### Technical risks

- [ ] **Field name drift between init and draft**. `slug_prefix:` must match exactly. Mitigation: SPEC defines the name as canonical contract (Phase 1 done first).
- [ ] **`audience-ops/projects/` naming collision with v0.11.x legacy semantics**. Mitigation: SPEC §4 conventions explicit warning + CLAUDE.md note.
- [ ] **YAML break from new content**. Mitigation: pre-tag YAML hard gate (Task 5.2).
- [ ] **Dossier read happens silently in draft → users don't realize it's active**. Mitigation: documented in README's capture-pattern subsection + in draft's Paso 2 wording.
- [ ] **Roberto stops maintaining dossiers after 2 weeks**. This is the dogfooding gate. Documented in PRD risks; revisit in v0.4.x if dead weight.

### Non-risks (explicitly considered, not problems)

- **`init` flow becomes too long**: Paso 7 is opt-in (one yes/no question; declining ends the sub-flow immediately). Adds ~30 seconds for users who decline; ~5-10 min for users with 3-4 projects.
- **Breaking v0.2.0 instances**: zero changes apply if `audience-ops/projects/` doesn't exist. Backwards compatible by design.
- **skills.sh.json description change**: no — dossier is internal to one instance, doesn't change the system's positioning.
