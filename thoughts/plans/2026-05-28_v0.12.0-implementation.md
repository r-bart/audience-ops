Breaking change owned explicitly. 13 files touched: SPEC.md (major), config.yaml, portfolio.yaml (deleted), 5 SKILL.md (init drastically, others path-updated), README + AGENTS + CLAUDE, .gitignore.

## Approach Analysis

### Option A: Contract-first (SPEC → data → skills → docs → release)

**Description**: SPEC.md establishes the new contract. Then config.yaml + delete portfolio.yaml encode the data shape. Then 5 SKILL.md follow the new contract. Then docs explain. Then release.

**Pros**:
- Each subsequent phase has a stable reference (SPEC) to align against.
- Reduces "is this path right?" guesswork during skill edits.
- Mirrors how the system was originally built (SPEC then skills).

**Cons**:
- SPEC.md is one big edit; if a section is wrong, all downstream tasks consume the error.
- Slightly slower than parallelizing everything from minute one.

**Complexity**: Medium.

### Option B: Parallel-first (multi-file edits at once, reconcile at end)

**Description**: Edit SPEC, config.yaml, all 5 SKILL.md, and docs concurrently. Reconcile inconsistencies in a final pass.

**Pros**: Fastest.
**Cons**: Inconsistency risk is HIGH for a refactor of this magnitude. Path mistakes propagate silently because there's no source of truth being established first.

**Complexity**: Medium-High.

### Recommendation

**Option A**. The breaking-change-refactor pattern needs a stable contract before downstream edits. Mirrors how Roberto built the system originally (SPEC then skills). Aligns with how v0.10.1 and v0.11.1 patches were structured (define → execute → review). Speed gained from Option B is offset by drift risk this strict-mode is designed to catch.

#### Sugerencias por defecto de formato según `id`

- **x**: Threads de 5–7 tweets; primer tweet = hook concreto; una idea por tweet; sin numerar.
- **linkedin**: Posts de 1200–1500 caracteres; primera línea = hook; párrafos cortos.
- **blog**: Long-form. Estructura: hook → tesis → desarrollo → cierre. 800–1500 palabras.
- **newsletter**: Subject ≤50 chars sin clickbait; preheader complementario; hook (2-3 frases) → cuerpo (1 idea) → CTA opcional; 400–700 palabras. **Importante**: requiere campos `subject` y `preheader` en frontmatter de cada publicación.
- **youtube**: Hook en los primeros 5s; estructura clara; CTA al final.
- **reddit**: Adaptarse al subreddit; sin auto-promoción explícita; valor primero.
